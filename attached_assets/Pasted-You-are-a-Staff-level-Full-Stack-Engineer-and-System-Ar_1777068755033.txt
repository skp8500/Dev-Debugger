PROBLEM: Developers waste hours debugging cryptic errors. Juniors especially 
get stuck reading stack traces they don't understand. No tool explains errors 
in plain language AND provides corrected code in one place.

SOLUTION: A clean, fast, AI-powered web app where any developer pastes broken 
code + error → gets a structured, expert-level debug report in seconds.

TARGET USERS:
  - Junior developers (primary) — need plain-English explanations
  - Students — need teaching, not just fixes
  - Senior devs — need speed, want corrected code instantly

CORE USER FLOW:
  1. Paste code + error message/stack trace
  2. Select language
  3. Hit "Debug" 
  4. Receive: Root Cause → Fix → Explanation → Corrected Code
  5. Copy/download result or revisit from history

  Why: SSR, file-based routing, API routes, production-ready out of the box
  - Tailwind CSS (utility-first, no CSS files bloat)
  - Monaco Editor (same engine as VS Code — professional feel)
  - React Query / TanStack Query (server state, caching, loading states)
  - Axios with interceptors (centralized error handling)
  - Zustand (lightweight global state — theme, history panel)
  - next-themes (dark/light mode with zero flash)
  - Framer Motion (purposeful animations only — not decorative noise)
  - React Hot Toast (non-intrusive notifications)

BACKEND — FastAPI (Python)
  Why: Async-native, automatic OpenAPI docs, pydantic validation, fastest 
  Python framework — ideal for AI workloads
  - Pydantic v2 (request/response validation with strict types)
  - SQLAlchemy 2.0 + asyncpg (async ORM, no blocking DB calls)
  - Alembic (database migrations — never raw SQL schema changes)
  - Redis + Celery (async AI job queue — non-blocking UX)
  - slowapi (rate limiting per IP)
  - structlog (structured JSON logging — production standard)
  - python-dotenv + pydantic Settings (config management)

AI LAYER
  - OpenAI GPT-4o (primary) with fallback to GPT-3.5-turbo
  - Streaming responses via SSE (Server-Sent Events) — UX feels instant
  - Prompt versioning (store prompts as constants, not inline strings)
  - Token usage tracking per request (stored in DB for analytics)

DATABASE — PostgreSQL 16
  Tables:
    users (id, created_at, ip_hash)          ← privacy-safe user tracking
    debug_sessions (id, user_id, language, raw_code, raw_error, 
                    root_cause, fix, explanation, corrected_code, 
                    tokens_used, duration_ms, created_at)
    prompt_versions (id, version, system_prompt, created_at)

CACHE — Redis 7
  - Cache identical {code + error + language} hash → response (TTL: 1 hour)
  - Rate limit counters per IP
  - Celery broker for async job queue

BASE URL: /api/v1

── POST /analyze ───────────────────────────────────────
Request body (validated by Pydantic):
{
  "code": "string (max 10,000 chars, required)",
  "error": "string (max 5,000 chars, required)",  
  "language": "python|javascript|typescript|c++|java|go|rust|php|ruby|csharp",
  "mode": "standard|eli5",   ← eli5 = Explain Like I'm 5
  "stream": true             ← enable SSE streaming
}

Success response (non-streaming):
{
  "session_id": "uuid",
  "root_cause": "string",
  "fix": "string",
  "explanation": "string",
  "corrected_code": "string",
  "language": "string",
  "severity": "syntax_error|runtime_error|logic_error|warning",
  "tokens_used": 847,
  "cached": false,
  "created_at": "ISO8601"
}

Error response (consistent shape always):
{
  "error": {
    "code": "RATE_LIMIT_EXCEEDED | INVALID_INPUT | AI_UNAVAILABLE | ...",
    "message": "Human-readable message",
    "details": {}  ← optional extra context
  }
}

── GET /history?page=1&limit=10&language=python ────────
Response:
{
  "sessions": [...],   ← array of session summaries
  "total": 47,
  "page": 1,
  "limit": 10,
  "has_next": true
}

── GET /history/:session_id ────────────────────────────
Returns full session object (same shape as /analyze response)

── GET /health ─────────────────────────────────────────
Returns: { "status": "ok", "db": "ok", "redis": "ok", "ai": "ok" }

SYSTEM PROMPT (stored in prompt_templates.py, version-tracked):

"""
You are a world-class programming mentor embedded in a debugging tool.
You combine the precision of a compiler with the clarity of a great teacher.

Your only job is to analyze broken code and return a structured debug report.

STRICT RULES:
- Never be vague. Every statement must be specific and actionable.
- Never say "it depends" without immediately explaining what it depends on.
- Never skip the corrected_code — always provide working, complete code.
- Never add pleasantries ("Great question!", "Sure!", "Of course!").
- If the error is genuinely ambiguous, state exactly WHY and list possibilities.
- Always detect the severity: syntax_error | runtime_error | logic_error | warning

RESPONSE FORMAT — return ONLY valid JSON, no markdown, no extra text:
{
  "root_cause": "One precise sentence naming the exact error and why it occurs",
  "severity": "syntax_error | runtime_error | logic_error | warning",
  "fix": "Step-by-step fix instructions. Number each step. Be specific.",
  "explanation": "Plain English explanation. No jargon. Imagine the reader is smart but new to programming.",
  "corrected_code": "Full corrected code. No truncation. No '...' placeholders. Complete and runnable.",
  "pro_tip": "One advanced insight a senior dev would add — a pattern, a best practice, or a warning about similar future errors"
}
"""

ELI5 MODE — appended when mode == "eli5":
"""
IMPORTANT: For the 'explanation' field only — explain as if talking to a 
10-year-old who has never seen code. Use a real-world analogy. 
Example: "Think of a variable like a labeled box — your code tried to open a 
box that doesn't exist yet."
Keep it fun, warm, and confidence-building. Make them feel smart, not dumb.
"""

USER PROMPT (constructed dynamically):
"""
Language: {language}
Error/Stack Trace:
{error}

Code:
{code}

Analyze this and return the JSON debug report.
"""

Frontend:
  - Axios interceptor catches all 4xx/5xx, maps to user-friendly messages
  - No raw "Internal Server Error" ever shown to user
  - Retry UI for transient failures
  - Offline detection with banner

Backend:
  - Global FastAPI exception handler → always returns consistent JSON shape
  - OpenAI timeout: 30s hard limit
  - If OpenAI returns malformed JSON: log it, return fallback error
  - DB connection failures: return 503, don't crash the server
  - All exceptions logged with request_id for traceability
